Democratizing the building of basic crowdsourcing projects

No need to deploy a server or a database to create your own micro-task application. Build your project by creating or remixing existing micro-Task template code (html/javascript) and embed it on your own web page.

Overview

What is a micro-contribution?
Micro-contributions are contributions of any kind that someone could quickly make remotely (or not) within a small amount of time. A micro-application manages micro-contributions. If we think about the diversity of contributions in participatory projects as a long tail with few core people and a large majority of volatile users,the goal of these micro-applications is to tap into this segment assuming that the sum of these micro-contributions could leverage a subtantial work.

What kind of micro-contributions?
e.g. analyzing a large flow of individual tweets, classifying a large volume of images, analyzing a large area on a map, segmenting very quickly a long video collaboratively, generating stories via the writing of individual sentences , or any kind of big data difficulty processable/generable by the machine but cutable in smaller parts and processable by volatile participants.

What is MechanicalTask?
MechanicalTask is a cloud toolkit, an attempt to democratize the development of crowdsourcing projects to anyone having some basic knowledge in html/javascript.

Features

  • Embed your micro-app like a youtube video

    MechanicalTask is not a platform hosting crowdsourcing projects. It is also not a framework for coding applications. It's a web service generating micro-applications embeddable on your website like a youtube video, or for mobile phones, without hosting directly its infrastructure. So no need to deploy a server to setup a basic crowdsourcing project. Once embedded on a website, users/visitors can start completing some tasks. (note: you are responsible to host any required tutorials/doc on your website).

  • White box and data openess

    MechanicalTask is a distributed infrastructure using google fusion tables as database to store your tasks and the answers of volunteers. We use google fusion and not an internal database because it enables any third party to view, export, analyze, visualize the raw answers or event insert dynamically new tasks via a user interface or an API in a more transparent way. MechanicalTask is a kind of search engine indexing a first fusion table representing the stream of your micro-tasks to process, manages the allocation of tasks (with a given redundancy) according to your users and send the results to another fusion table (output).At any time everyone can see the raw data.

  • Coding in the cloud

    You are in charge of the user interface to collect the data via an HTML/javascript template. But no need to update your code locally and deploy it on MechanicalTask. You can code and modify the UI of an app on the fly via an online editor.

  • Remix culture and micro-task ecosystem

    What is valuable is not only the data but also the mechanism you used to collect them, engage people. So if you generate a project via this platform your micro-app template will be publicly accessible with the community. MechanicalTask uses Gist as a first place for sharing your code.So newcomer can remix, fork it to build their own micro-apps.
    Such openess will facilitate the emergence of eco-system with best practices in micro-tasking, and leverage more quickly a potential community.

Demo of web widgets

Images classification

A common toy example is to classify a large amount of photos. In this example we assume you have csv or xsl spreadsheets with all the url of the images to classify. Each row is a task with for instance the url of the image to classify. Once this table imported on a google fusion table (you can also have a programm filling/updating the fusion table to add new tasks/images), you need to add the ID of the google fusion and create the UI Template to generate the web widget and the output table.

  • Tasks input table with a list of images and other contextual information

  • Generated web widget to classify images.

Tagging Images

A similar example is about annotating large amount of images e.g. tagging galaxies like GalaxyZoo. In this case the input table is the same as for the previous example about classifying images. Only the instructions , and tools to process the task change. So only the UI template needs to be updated.

Analyzing a large area

Let assume you want to create quickly a crowdmapping project asking to volunteers to identify objects inside a large area.
  • First you need to create a fusion table representing the micro-tasks to process. You can generate this table yourself. we also propose you a task generator cutting your map into a grid of smaller areas, easier to analyze individually. The result is sent to a google fusion table. You need to set up the permission of the fusion table so it is publicly visible if it is not already done.
  • >Then you need create a crowdsourcing project. first add the encrypted ID of the fusion table so MechanicalTask can index the table (without understanding that is about mapping).
  • As you are responsible of the interpretation of the tasks e.g the UI and the information you want to collect, you also must add a script template. This template can be copied and modified from other projects. This will generate a web widget and another fusion table to store the answers of volunteers.
  • Finally you can embed the web widget in your website and add further information (e.g. tutorial) if required.
  • At any time you have visualize and export the answers via the google fusion.

  • Task generator helping you to divide the analysis of a large area

  • GeoVisualisation (using the google fusion tools) of the generated input google fusion table with a lots of small areas

  • Generated web widget you can embed on your web site

  • GeoVisualization (using the google fusion tools) of the Generated output fusion table storing the answers
